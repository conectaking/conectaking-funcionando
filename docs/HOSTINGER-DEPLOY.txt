================================================================================
CONECTA KING — Deploy na Hostinger (front-end em public_html)
================================================================================
(Ficheiro versionado no Git. Na tua máquina, a pasta public_html espelha isto.)

0) O QUE NÃO É O PROBLEMA
   O back-end (Node no Render / conectaking-api) NÃO serve conectaking.com.br.
   Quem serve o site é só a Hostinger: ficheiros em public_html + .htaccess + PHP.
   Alterações na API não alteram o teu hosting até fazeres deploy de ficheiros lá.

0c) ONDE VIVE O PAINEL (dashboard.html) — NÃO confundir com “subir para o back-end”
   - No projeto Git, o ficheiro que editas costuma estar em **public/dashboard.html**
     (código-fonte / referência no repositório).
   - O comando **npm run sync:king-docs** só corre **na tua máquina**: copia alguns
     ficheiros de **public/** para a pasta **public_html/** para preparares o pacote
     que vais **enviar por FTP/File Manager à Hostinger**. Não envia nada ao Render
     nem substitui o deploy da API.
   - O painel que os clientes abrem em **conectaking.com.br** é sempre o que está
     na **Hostinger** (o public_html do servidor). Se não subires o dashboard.html
     novo para lá, o site continua com a versão antiga — por isso às vezes parece
     que “não atualiza”: falta o upload ao hosting, não falta o back-end.
   - O servidor Node no Render **pode** ter uma rota que também serve um
     /dashboard.html a partir de public/ (útil em testes ou URL da API). Isso
     **não é** o teu site principal na Hostinger; não dependas disso para o domínio
     do cliente.

0b) .HTACCESS MÍNIMO (recuperar o site)
   O public_html/.htaccess foi reduzido ao MÍNIMO (só DirectoryIndex + UTF-8), sem
   mod_rewrite nem headers, para não bloquear login, dashboard nem o resto.
   Abre sempre com URL completa: /login.html, /dashboard.html, etc.
   Regras opcionais (URLs limpas /kingSelection) estão em htaccess-regras-opcionais.txt
   para ativares só quando o site já estiver estável.

1) RAIZ DO DOMÍNIO (causa n.º 1 do 403 na raiz /)
   O domínio TEM de usar como "Raiz do documento" a pasta public_html onde estão
   index.html, login.html, .htaccess (não a pasta "pai" onde só aparece a pasta
   public_html vazia por fora).
   Passo a passo: docs/HOSTINGER-RAIZ-PUBLIC-HTML.md
   Não há como "forçar isto" pelo código do back-end — só pelo hPanel Hostinger
   (ou suporte a dizer: document root = public_html com os teus ficheiros).

2) FICHEIROS
   Envie TODO o conteúdo da pasta public_html (no teu PC) para essa raiz no
   servidor. Inclua ficheiros ocultos: .htaccess

2b) KING DOCS (kingDocs.html)
   O site na Hostinger lê public_html/, NÃO a pasta public/ do back-end no Git.
   Depois de alterar public/kingDocs.html ou public/kingDocsShare.html no projeto,
   corre na raiz do repo:  npm run sync:king-docs
   Isto copia os dois ficheiros, public/dashboard.html → public_html/dashboard.html
   e também public/js/landing-exit.js, dashboard-cropper-enhance.js e image-crop-modal.js → public_html/js/.
   Volta a enviar public_html/kingDocs.html (e kingDocsShare.html se mudaste) para o hosting.

2c) SAIR NA PÁGINA INICIAL (index.html)
   Copie public/js/landing-exit.js para public_html/js/landing-exit.js (upload ou espelho local).
   No index.html, antes de </body>, adicione uma linha:
     <script src="js/landing-exit.js?v=2026-04-23" defer></script>
   Com sessão (token no navegador), o script mostra o botão "Sair" ao lado de "Acessar Painel"
   (deteção automática). Se o layout for muito diferente, coloque no header:
     <span id="landing-sair-slot"></span>
   e o "Sair" aparece dentro desse elemento.

3) PASTA kingSelection VAZIA (causa de 404 em /kingSelection/)
   Se existir uma pasta chamada "kingSelection" vazia (ou sem index) dentro de
   public_html, apague-a. O Apache pode servir essa pasta em vez do rewrite e
   dar 404. O rewrite envia /kingSelection para kingSelectionEdit.html (ficheiro).

4) .HTACCESS + index.php
   public_html/.htaccess = versão MÍNIMA (DirectoryIndex + UTF-8).
   public_html/index.php = fallback se o servidor pedir index.php; envia o mesmo
   conteúdo que index.html quando este existe.

5) TESTES RÁPIDOS (após upload)
   https://www.conectaking.com.br/index.html
   https://www.conectaking.com.br/kingSelectionEdit.html
   https://www.conectaking.com.br/kingSelection
   Se os dois primeiros funcionarem e /kingSelection não, é rewrite ou pasta vazia.

6) ERRO 500 APÓS SUBIR .HTACCESS
   Renomeia temporariamente .htaccess para .htaccess.bak. Se o site voltar,
   contacta o suporte Hostinger (mod_rewrite).

7) API
   config.js → conectaking-api.onrender.com (independente do hosting).
